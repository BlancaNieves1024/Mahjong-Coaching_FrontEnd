## 牌譜分析画面 プロトタイプ開発進捗・報告事項

### 1. 報告すべき技術仕様（コードのアーキテクチャ）

開発した仕組みの意図と設計思想について説明します。

* **データ処理の分離**
* 抽出（`extract.py`）と可視化（`main.py`）の処理を分離。将来的なバックエンドの実装（API化）を見据えた柔軟な構造にしています。


* **JSON連携の標準化**
* 中間ファイルとして `data.json` を介する形式を採用。バックエンド完成時のAPI統合を容易にする設計です。


* **テンプレートエンジンの採用**
* `Jinja2` を活用し、Pythonのデータを直接HTMLへレンダリング。JavaScriptによる複雑なDOM操作を排除し、レンダリング速度とメンテナンス性を向上させました。


* **牌コードの正規化**
* システム内部の牌コード（`4m`等）と、UI表示用の牌記号（Unicode/日本語）を分離し、表示のカスタマイズ性を確保しています。



### 2. バックエンドチームに連携すべき内容

今後の開発に向けた仕様のすり合わせ事項です。

* **JSONのデータ構造の合意**
* 現在定義しているJSON構造（`kyoku`, `turn`, `tehai`, `loss` 等）をベースに、APIのレスポンス仕様を確定させたいと考えています。


* **牌コードの定義の統一**
* バックエンドから出力される牌コードの命名規則を、現在プロトタイプで定義しているリスト（`4m`, `1s` 等）に統一をお願いします。


* **期待値ロス（loss）の計算基準**
* `loss = 0` の場合に「完璧（Perfect）」判定を行う仕様です。バックエンド側での計算精度や、Perfectとする閾値（許容範囲）について定義の確認が必要です。


* **将来的なAPI化への接続**
* 現状は `data.json` を介していますが、今後はREST APIのJSONレスポンスを直接受け取れるようにする予定です。その際の連携インターフェースを事前に共有したいと考えています。

### 大脇への回答(Javascriptは使わないのか？)

* **段階的アプローチの意図**
* 現段階はデータ構造とUI連動の検証を最優先するため、Pythonテンプレートによる静的構成を選択しました。


* **役割分担の明確化**
* バリデーションや動的なエラーチェックはフロントエンド（React）、データの整合性やルール判定はバックエンドと、明確に役割を分ける設計を予定しています。


* **開発効率の観点**
* まずプロトタイプで仕様を固めた後に、Reactへコンポーネントとして移植する方針をとることで、手戻りを最小限に抑える狙いがあります。
